## 今日の一言
Loop engineering は「良いプロンプトを書く技術」から「エージェントが止まり、検証され、再利用可能に学ぶループを設計する技術」へ、実務記事と学術研究の両側で急速に輪郭が固まっています。

### 1. Loop Engineering徹底解説：自律エージェントを「動く」から「運用できる」へ
- 出典: Qiita / Web記事
- 日付: 2026-07-10（Bing RSS確認）
- リンク: https://qiita.com/Simon_Zhang/items/388cd1125860591f5c8d
- 要約: Loop Engineering を、AIエージェントがタスク達成まで繰り返す制御ループを信頼性・安全性・コスト・観測性の観点から設計する技術として整理している。単なる自動実行ではなく、停止条件、検証、状態管理、ログ、再試行を含む「運用設計」へ焦点が移っている点が重要。
- なぜ面白いか:
  - 技術: ループをプロンプト列ではなく、状態遷移・評価器・ガードレール・監視を持つ実行システムとして扱うため、実務導入時の失敗モードを議論しやすい。
  - 人文: ethics の観点では、自律性を高めるほど「誰が止めるのか」「どの外部副作用まで許すのか」という責任境界が中心問題になる。anthropology 的には、開発者の仕事が命令者から制度設計者・儀式設計者へ変わる兆候として読める。

### 2. Loop Engineering入門：AIコーディングエージェントを動かすループ設計
- 出典: Zenn / Web記事
- 日付: 2026-07-12（Bing RSS確認。URL上は 2026-06-10）
- リンク: https://zenn.dev/suwash/articles/loop-engineering_20260610
- 要約: 設計対象を「個々のプロンプト」から「ループ（システム）」へ移す入門記事。人間が毎ターン指示するのではなく、発見・割当・実行・検証・再実行を回す仕組みを設計するという発想を日本語で分かりやすく示している。
- なぜ面白いか:
  - 技術: コーディングエージェントの生産性を、モデル単体の賢さではなく、タスク発見や検証を含む周辺ループの品質で評価する視点を提供する。
  - 人文: history の観点では、職人が道具を直接操作する段階から、工程表・品質管理・現場監督を設計する段階へ移った産業史と似ている。creativity の観点では、人間の創造性が一文の命令から「どんな反復構造なら良い成果が育つか」の設計に移る。

### 3. Loop Engineering Guide: Build Safe Autonomous Agent Loops
- 出典: Webガイド
- 日付: 2026-07-11（Bing RSS確認）
- リンク: https://loopengineering.run/
- 要約: 「手で一回ずつプロンプトを与える」のではなく、作業発見、エージェントへの割当、成果検証、状態保存、スケジュールまたはトリガーによる再実行までを含む安全な自律ループ構築ガイド。英語圏向けに、実装パターンとしての loop engineering を簡潔に定義している。
- なぜ面白いか:
  - 技術: discover → assign → verify → persist → rerun という一連のパイプラインとして抽象化することで、cron、CI、キュー、評価器を接続した実装に落とし込みやすい。
  - 人文: philosophy 的には、エージェントの「意図」をモデル内部ではなく外部の反復構造に宿す見方であり、行為主体性をシステム全体に分散させる考え方に近い。ethics 的には、安全性を個別回答の検閲ではなく、継続的な検証と停止可能性として設計する点が示唆的。

### 4. When Agents Do Not Stop: Uncovering Infinite Agentic Loops in LLM Agents
- 出典: arXiv / 論文
- 日付: 2026-07-02
- リンク: https://arxiv.org/abs/2607.01641
- 要約: LLMエージェントが計画、ツール使用、状態更新、エージェント間ハンドオフを繰り返す中で、終了条件が実効的に働かず「Infinite Agentic Loops（IALs）」に陥る問題を定式化している。IAL-Scan という静的解析ツールで、実プロジェクト中の無限ループ失敗を検出する方向を示す。
- なぜ面白いか:
  - 技術: loop engineering の最重要リスクである「止まらないエージェント」を、コスト爆発、モデルDoS、コンテキスト肥大、外部副作用の反復として分析可能なソフトウェア工学問題にしている。
  - 人文: ethics の観点では、AIの危険性が「悪い意図」だけでなく、善意の自動化が止まらない制度設計からも生じることを示す。narrative 的には、魔法使いの弟子型の古典的物語が、API課金とツール実行の現代的リスクとして再演されている。

### 5. TTHE: Test-Time Harness Evolution
- 出典: arXiv / 論文
- 日付: 2026-07-09
- リンク: https://arxiv.org/abs/2607.08124
- 要約: LLMエージェントの挙動はモデルだけでなく、コンテキスト構築、ツール呼び出し、中間検証、失敗回復を担うハーネスに大きく左右されるとし、テスト時の実行トレースだけを使ってハーネス自体を進化させる Test-Time Harness Evolution を提案している。固定ワークフローではなく、失敗モードに応じてループの外枠を適応させる研究。
- なぜ面白いか:
  - 技術: loop engineering を手作業の設計論から、実行トレースを使って検証器・リカバリ・ツール接続を改善する自己適応ハーネス研究へ押し広げている。
  - 人文: philosophy 的には、知能を「答える能力」ではなく、失敗から環境との関係を組み替える能力として捉える点が面白い。creativity の観点では、作品を一度で完成させるのではなく、制作プロセスそのものを変形させるメタ創作に近い。
