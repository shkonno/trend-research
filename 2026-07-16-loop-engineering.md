## 今日の一言
Loop engineering は「良いプロンプトを書く」から「検証・停止・記憶・人間承認を含む反復システムを設計する」へ、エージェント時代の実務語彙として急速に固まりつつあります。

## トップ5

### 1. Claude Code文脈で `/goal`・`/loop`・`/schedule` がループ設計の型として共有される
- 出典: X投稿（cc_lab_jp、vibelancer、realsanb などの関連投稿）
- 日付: 2026-07-13〜2026-07-15頃
- リンク: https://x.com/cc_lab_jp/status/2077153814314971411
- 要約: Claude Codeまわりで、明確な成功条件まで走る `/goal`、セッション中に周期実行する `/loop`、クラウド上で持続する `/schedule` という分類が広く共有されています。話題の中心は、単発プロンプトではなく、停止条件・検証者・状態保存・スケジュールを含む「エージェント運用の設計」へ移っています。
- なぜ面白いか:
  - 技術: ループを「内側の目標達成」「外側の定期実行」「永続スケジュール」に分けることで、エージェントの暴走や無限リトライを実装レベルで抑えやすくなります。
  - 人文: philosophy/ethics の観点では、これはAIを“意図を推測する相手”ではなく“制約された制度の中で働く主体”として扱う転換です。人間の責任は、命令文の巧拙から、どの条件で止め、誰が承認し、失敗をどう記録するかという統治設計へ移ります。

### 2. LOGOS論文が「verifiable human-agent loop engineering」を提示
- 出典: arXiv / X投稿 / 富士通Web記事
- 日付: 2026-07-12（arXiv公開）、2026-07-14頃にXで話題化
- リンク: https://arxiv.org/abs/2607.10878
- 要約: arXiv論文「LOGOS: A Living Logic for AI Agent Teams That Evolve With Humans」は、マルチエージェントの自己改善を、証跡・権限・テスト・人間承認つきの“昇格候補”として扱う枠組みを提案しています。富士通の関連ページでも、業務知識・実行結果・人のフィードバックをもとにマルチAIエージェントシステムを継続改善する方向性が説明されています（https://global.fujitsu/ja-jp/technology/research/article/topics/202607-multi-aI-agent-framework）。
- なぜ面白いか:
  - 技術: 学習したプロンプト、メモリ、ツール、役割、ワークフローを即時反映せず、検証証拠・ポリシー・明示承認を通過したものだけ昇格させる fail-closed な設計が中心です。
  - 人文: ethics/history の観点では、これは工場の品質保証や変更管理をエージェントの自己進化に移植する発想です。「提案」と「本番昇格」を分けることで、AIの自律性を拡張しながらも、制度的な責任の所在を人間側に残そうとしています。

### 3. Andrew Ngの3層フィードバックループが「何を作るか」を変える議論として再流通
- 出典: X投稿（Andrew Ngの枠組みを引用・要約する関連投稿）
- 日付: 元投稿は2026年6月下旬〜7月初旬に流通、2026-07-02以降も継続的に議論（古いが重要）
- リンク: https://x.com/flingjie/status/2072670394205757609
- 要約: Andrew Ngの文脈では、分単位の agentic coding loop、時間単位の developer feedback loop、日〜週単位の external feedback loop が並走する構図が注目されています。AIがコードを書いてテストする内側の速度だけでなく、人間がプロダクト判断を行い、外部ユーザーから学ぶ外側のループが価値を決める、という整理です。
- なぜ面白いか:
  - 技術: テスト・評価・ブラウザ確認などで内側の自動化を高速化しつつ、外側にユーザー検証と仕様更新のループを置くことで、コード生成をプロダクト学習に接続できます。
  - 人文: anthropology/narrative の観点では、開発者の物語上の役割が「実装者」から「複数の時間軸を調停する編集者」へ変わります。AIが速く作るほど、人間には“何を作らないか”を選ぶ文化的・文脈的判断が強く求められます。

### 4. LMCache + vLLM と RAG最適化ループが、コスト削減と精度改善の実装テーマに接続
- 出典: X投稿（ai_hakase_、lmcache などの関連投稿）
- 日付: 2026-07-13〜2026-07-14頃
- リンク: https://x.com/ai_hakase_/status/2076894963556548973
- 要約: LMCacheのvLLMレシピ統合やKVキャッシュ再利用の話題と、RAGのチャンクサイズ・埋め込み・検索パラメータを目標recallまで反復調整するLoop Engineeringが結びつけられています。エージェントやRAGの長文・多ターン処理で、同じ文脈を何度もprefillしないことがコスト面の焦点になっています。
- なぜ面白いか:
  - 技術: ループ設計が「推論を何回回すか」だけでなく、KVキャッシュ、vLLM、RAG評価指標まで含むシステム最適化問題になっています。
  - 人文: history/ethics の観点では、自動化の普及は常に“計算資源を誰が払うか”という経済史的問題を伴います。Loop engineering がコスト削減と結びつくことで、AI活用の格差を広げるのか縮めるのかという実務的倫理が前面に出ます。
