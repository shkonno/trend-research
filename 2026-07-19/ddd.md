## 今日の一言
AIエージェント時代のDDDは「モデルを作る手法」から、仕様・判断・責任をどこに置くかを可視化する組織的な設計リテラシーへ広がっている。

## トップ5

### 1. ddd-agent-skill: DDDをエージェントスキルとして実装する試み
- 出典: GitHubリポジトリ
- 日付: 2026-07-07 更新
- リンク: https://github.com/aristorinjuang/ddd-agent-skill
- 要約: Vaughn Vernon の IDDD に沿い、DDDが本当に必要かを判定する「worthiness gate」から始め、イベントストーミング、ユビキタス言語、境界づけられたコンテキスト、集約設計までをエージェントの段階的ワークフローに落とし込むプロジェクト。Claude Code、Cursor、Codex、Copilot、Windsurf、Gemini など Agent Skills 互換環境での利用を想定している。
- なぜ面白いか:
  - 技術: DDDの実践手順を、単なるコード生成プロンプトではなく、質問・判定・モデリング・監査を含む再利用可能なエージェント能力としてパッケージ化している。
  - 人文: 「DDDをやるべきか」を先に問う設計は、手法信仰ではなく、チームの認知負荷や事業上の複雑さに応じて適切な儀式を選ぶ文化を促す。ユビキタス言語を人間とAIの共有語彙にする点も、設計を対話の制度として捉え直している。

### 2. Decision-Driven Design: LLM時代の「仕様保存則」
- 出典: GitHubリポジトリ
- 日付: 2026-07-10 更新（2026-07-14 push）
- リンク: https://github.com/Hafeok/decision-driven-design
- 要約: LLMシステムの信頼性を、仕様が「上流のエンコード」「機械的検証」「人間の判断」「欠陥としての流出」のどこに配分されるかで説明する設計フレームワーク。DDDそのものではないが、ドメイン文脈を与えられたLLMが判断を行うとき、仕様の不足は消えずに場所を変えるだけだという主張が、AI時代の設計議論に強く接続している。
- なぜ面白いか:
  - 技術: プロンプト、スキーマ、コンテキスト、検証、レビューを同じ「仕様需要」の配分問題として扱い、LLMエージェント設計の品質会話を監査可能な形にしている。
  - 人文: 「仕様を省いた分はユーザーに欠陥リスクとして渡される」という見方は、設計をコスト削減ではなく責任配分の倫理として読ませる。DDDのユビキタス言語が、ここでは組織内の約束と説明責任を支える語彙になる。

### 3. AppFactory: DDD、GraphQL、エージェントを組み合わせたAIネイティブ開発工場
- 出典: GitHubリポジトリ
- 日付: 2026-07-17 更新
- リンク: https://github.com/antoniofregoso/AppFactory
- 要約: 小規模チームがエンタープライズ級アプリケーションを構築するためのAIネイティブなソフトウェアファクトリを掲げ、DDD、GraphQL、再利用可能アーキテクチャ、知的エージェント、コード生成を組み合わせている。自然言語検索を型付き検索計画へ変換するなど、業務語彙と実装をつなぐ仕組みも含む。
- なぜ面白いか:
  - 技術: DDDを単独のモデリング手法ではなく、宣言的UI、GraphQL、コード生成、AI検索と一体化したプロダクト生成基盤の中核に置いている。
  - 人文: 「小さなチームを高性能な組織にする」という語りは、AIが人員を置き換える話ではなく、ドメイン理解を共有資産化することで組織能力を拡張する話として読める。DDDがチームの言葉をソフトウェア工場の入力に変換する媒介になっている。

### 4. AI Project Manager: DDD/CQRS/マルチエージェントによる監査可能なAIオーケストレーション
- 出典: GitHubリポジトリ
- 日付: 2026-07-13 更新
- リンク: https://github.com/MUKHTIARSHAH/AI-Project-Manager
- 要約: ASP.NET Core、Clean Architecture、DDD、CQRS、MediatR、イベント駆動アーキテクチャ、マルチエージェントAIを組み合わせたエンタープライズ向けAIプロジェクト管理基盤。READMEでは、人間の承認、AIエージェント、ソフトウェアデリバリーを監査可能なガバナンスで調整することを目的としている。
- なぜ面白いか:
  - 技術: DDD/CQRS/イベント駆動を、AIエージェントのワークフロー制御と人間承認を扱う基盤アーキテクチャに接続している。
  - 人文: エージェントが増えるほど、誰が何を承認し、どの判断がどの文脈で行われたのかを残す社会的な仕組みが重要になる。DDDの境界づけられたコンテキストは、技術境界だけでなく、責任と権限の境界を設計する道具にもなる。

### 5. Automating Domain-Driven Design: Experience with a Prompting Framework
- 出典: arXiv
- 日付: 2026-03-27（古いが重要）
- リンク: https://arxiv.org/abs/2603.26244
- 要約: LLMとの構造化された対話で、ユビキタス言語の確立、イベントストーミングのシミュレーション、境界づけられたコンテキストの特定、集約設計、技術アーキテクチャへの写像というDDD活動を自動化しようとする論文。FTAPIの企業プラットフォーム要件を用いたケーススタディでは、初期の語彙整理やコンテキストマップは有用だった一方、後段の集約・技術設計では小さな誤りが蓄積して実用性を下げたと報告している。
- なぜ面白いか:
  - 技術: LLMはDDDを完全自動化する設計者ではなく、特に初期探索で有効な「スパーリング相手」として位置づけるのが妥当だという実証的な境界線を示している。
  - 人文: これはAIに設計判断を委譲する誘惑へのブレーキでもある。専門家が重要なトレードオフに集中するための道具としてAIを使う、という人間中心の設計文化を支持している。
