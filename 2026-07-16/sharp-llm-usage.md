## トップ5

### 1. Failure as a Process: CLIコーディングエージェント失敗の解剖
- 出典: arXiv論文 / X解説（cv_usk、DAIR.AIほか）
- 日付: 2026-07-10（arXiv公開）、2026-07-13頃にXで拡散
- リンク: https://arxiv.org/abs/2607.09510 / https://x.com/cv_usk/status/2076780727656751356
- 要約: CLI型コーディングエージェントの3,843軌跡から1,794件を精査し、失敗を「最終結果」ではなく、発生・進行・回復不能化のプロセスとして分析した研究。決定的エラーは中央値で7ステップ目に起き、誤った前提などの認識論的エラーが大きいという示唆が、実務者の「最初に前提を検証せよ」という議論と強く接続しています。
- なぜ面白いか:
  - 技術: 長いLLMワークフローでは、完了後レビューよりも序盤の前提検証、エラー信号への反応、巻き戻し可能なチェックポイント設計が効くことをデータで示しています。
  - 人文: これは「失敗」を個人やモデルの能力不足ではなく、時間の中で形成される認識のズレとして見る視点です。人間のプロジェクトでも、早い段階の思い込みが後工程の合理的な努力を全部空回りさせることがあり、AI運用はその縮図になっています。

### 2. Do AI Agents Know When a Task Is Simple? E3で過剰文脈を削る
- 出典: arXiv論文 / X投稿（mlexploration、asteris_aiほか）
- 日付: 2026-07-14
- リンク: https://arxiv.org/abs/2607.13034 / https://x.com/mlexploration/status/2077296327529214144
- 要約: 多くのエージェントは単純な修正にも最大文脈・大量ファイル読み込みで臨み、コストとノイズを増やす。論文はE3（Estimate, Execute, Expand）を提案し、まず最小十分な経路で実行し、検証に失敗したときだけ探索範囲を広げることで、成功率を保ちながらトークン91%削減、コスト85%削減、調査ファイル92%削減を報告しています。
- なぜ面白いか:
  - 技術: 「大きなコンテキストに全部入れる」ではなく、タスク難度を見積もり、検証失敗時だけ拡張する段階的実行が、実務の高速・低コストなエージェント設計に直結します。
  - 人文: これはAIに「全力で頑張る」ことではなく「どれだけ考えれば十分か」を教える話です。知的労働の価値が、努力量の最大化から注意の節約と節度へ移っていることをよく表しています。

### 3. From Prompts to Contracts: プロンプトではなく契約でLLMを囲う
- 出典: arXiv論文 / X投稿（fumisukekato、SciFiほか）
- 日付: 2026-07-09（arXiv公開）、2026-07-12頃にXで議論
- リンク: https://arxiv.org/abs/2607.08028 / https://x.com/fumisukekato/status/2076244400523596040
- 要約: 企業向けLLMエージェントを、プロンプト中心ではなく、ソースマニフェスト、出典付き主張、回答契約、検証ゲート、実行トレースで囲む「ハーネスエンジニアリング」として再構成する研究。モデルを入れ替えても、コード側が担う契約検証は270/270で保持されたと報告されています。
- なぜ面白いか:
  - 技術: 信頼性をモデルの善意やプロンプト文言に預けず、スキーマ・許可ソース・検証器・決定的フォールバックに移す設計は、RAGや社内エージェントの本番化にそのまま使えます。
  - 人文: 「AIに何を言わせるか」から「AIに何を言わせない構造を作るか」への転換です。組織でAIを使うとは、知性を解放するだけでなく、責任の所在と監査可能性を設計することでもあります。

### 4. AIに8割書かせたコード、半年運用の答え合わせ
- 出典: Zenn記事 / Xでの日本語圏拡散
- 日付: 2026-07-03頃にXで拡散（記事日付はWeb検索環境制約により本文取得未確認）
- リンク: https://zenn.dev/rapls/articles/7a1bac83df8b21
- 要約: AI生成コードを半年運用して、効いた習慣と腐った習慣を振り返る実践記事。残すべきは「なぜ」のコメント、小さな関数、人間が握る仕様とテストであり、腐りやすいのは大量コメント、早すぎる抽象化、「きれいに書いて」のような曖昧指示だという整理が注目されています。
- なぜ面白いか:
  - 技術: LLMコーディングでは、AIに足させるより人間が仕様・境界条件・テストを固定し、生成後に削る運用のほうが長期保守に強いという現場知を与えています。
  - 人文: 半年後に価値が残るかという時間軸を入れると、AIの「その場のもっともらしさ」は一気に疑わしくなります。創造性よりも保守、雄弁さよりも沈黙、追加よりも削除が重要になるのが、人間とAIの協働らしい逆説です。

### 5. Prompt / Context / Harness / Loop Engineering の4層モデル
- 出典: X投稿・スレッド（Avi Chawlaほか）
- 日付: 2026-07-03以降に継続議論
- リンク: https://x.com/_avichawla/status/2072980277870383366 / https://x.com/alex_prompter/status/2073462823179731032
- 要約: LLM活用を、プロンプト、コンテキスト、ハーネス、ループの4層で捉える整理がXで広く共有されています。単発入力を磨くプロンプトから、何を見せるかを設計するコンテキスト、ツール・検証・リトライを担うハーネス、停止条件と自律反復を担うループへ、実務の関心が外側へ移っています。
- なぜ面白いか:
  - 技術: 複雑なLLMワークフローを改善するとき、失敗箇所を「文言」「文脈」「実行基盤」「反復制御」に分解でき、対策を混同しにくくなります。
  - 人文: この4層モデルは、AIを魔法の会話相手ではなく、制度・環境・習慣の中で働く行為者として扱う見方です。人間の仕事もまた、才能だけでなく、机の上の資料、手順、レビュー、締切によって形作られることを思い出させます。
