## トップ5

### 1. Claude Security plugin for Claude Code beta: 端末内セキュリティスキャンと高コスト失敗例
- 出典: X投稿（Anthropic公式告知および利用者の失敗報告）
- 日付: 2026-07-22〜2026-07-23頃
- リンク: https://x.com/i/status/2079990597973057691 / https://x.com/i/status/2080080344481775690
- 要約: Claude Code用の「Claude Security」プラグインがベータ公開され、変更差分やコードベース全体を端末から脆弱性スキャンできる流れが話題になった。一方で、実利用者からは約600ドル相当のトークン消費後に分類器がセッションを壊したという報告もあり、能力と運用コストのギャップが鮮明になっている。
- なぜ面白いか:
  - 技術: コーディングエージェントを「生成」から「リスク検出・検証・修正提案」まで拡張する一方、推論コストとセッション信頼性がワークフロー全体の制約になることを示している。
  - 人文: これはAIを熟練者の補助者として信頼するための制度設計の話でもある。安全確認をAIに任せるほど、人間は「結果」だけでなく「検査にいくらかかり、どこで破綻したか」を読むリテラシーを求められる。

### 2. AgentDebugX: LLMエージェント失敗を Detect / Attribute / Recover / Rerun で閉ループ化
- 出典: arXiv / X投稿
- 日付: 2026-07-21
- リンク: https://arxiv.org/abs/2607.18754 / https://x.com/AINativeF/status/2080079394207678617
- 要約: AgentDebugXは、LLMエージェントの失敗を単なるトレース再生ではなく、検出・原因帰属・回復・再実行の閉ループとして扱うオープンソースデバッグ枠組み。中核のDeepDebugは、全体軌跡の理解、構造化された調査、クロスチェックによって、エラーが表面化したステップではなく根本原因を探すことを狙う。
- なぜ面白いか:
  - 技術: エージェント運用で最もつらい「どのステップが本当に悪かったのか」を、ログ閲覧から原因診断・修復まで引き上げようとしている。
  - 人文: 失敗を隠すのではなく、失敗の由来を共同で読む文化をつくる点が重要だ。AI活用が組織に入るほど、責任は「誰がミスしたか」から「どの連鎖を見逃したか」へ移る。

### 3. OpenWiki: コードベースや個人知識をエージェント向けWikiとして保守するCLI
- 出典: GitHub / X投稿
- 日付: GitHub更新確認 2026-07-23、Xでの紹介 2026-07-22〜2026-07-23頃
- リンク: https://github.com/langchain-ai/openwiki / https://x.com/3ryucom/status/2080075954802819174
- 要約: LangChainのOpenWikiは、コードベースやローカル知識ソースから、エージェントが参照しやすいWikiを生成・更新するCLI。READMEではコードモードと個人モードが説明され、GitHub ActionsなどのCIでドキュメント更新PRを出す運用も想定されている。
- なぜ面白いか:
  - 技術: LLMに毎回長大なコンテキストを渡すのではなく、コードや個人知識を継続的に整理された外部記憶へ変換する実践的な文脈管理パターンになっている。
  - 人文: 「記憶」を個人の頭やSlack検索に閉じず、機械と人間が共有できる読み物へ整える発想がある。これは組織の暗黙知を、AI時代の公共的なノートへ変える試みでもある。

### 4. Do AI Agents Know When a Task Is Simple?: 最大コンテキスト主義への警告
- 出典: arXiv
- 日付: 2026-07-14
- リンク: https://arxiv.org/abs/2607.13034
- 要約: この論文は、LLMエージェントが単純なタスクにも最大コンテキスト優先で臨み、既に読んだファイルや依存関係を再読してしまう問題を扱う。ワンライン修正が不必要に高コスト・高遅延なワークフローへ膨らむという、現場で起きがちな非効率を正面から捉えている。
- なぜ面白いか:
  - 技術: 「コンテキストを増やせば賢くなる」という直観に対し、タスク複雑度に応じて推論量・参照量を調整する必要性を示している。
  - 人文: 人間の熟練にも「大げさに考えない」技術がある。AIエージェントにも、全力疾走ではなく節度ある判断を教える段階に来ている。

### 5. Copy Less, Ground More: 長文コンテキスト推論における反復コピー失敗
- 出典: arXiv
- 日付: 2026-07-21
- リンク: https://arxiv.org/abs/2607.19345
- 要約: 長文コンテキストでステップバイステップ推論を行うLLMが、根拠を使う代わりに同じ内容を反復コピーしてしまう失敗モードを特定し、Evidence-Aware Reinforcement Learningで改善を狙う研究。長い資料を入れれば深く読める、という実務上の期待に対して重要な注意を与える。
- なぜ面白いか:
  - 技術: RAGや長文入力の品質を、引用量ではなく根拠への接地と反復抑制で評価すべきだと示唆している。
  - 人文: AIの「もっともらしい長文」は、理解ではなく模倣の過剰であることがある。読み手側も、流暢さではなく根拠との関係を検証する態度が必要になる。
