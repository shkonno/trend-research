## トップ5

### 1. Addy Osmani「Own the Outer Loop」— エンジニアは外側の責任ループを所有せよ

- 出典: Web記事 / Xでの共有
- 日付: 2026-07-15（記事公開） / X上の関連議論は2026-07-21前後に再拡散
- リンク: https://addyosmani.com/blog/own-the-outer-loop/
- 要約: Addy Osmaniは、AIエージェントが調査・実装・検証・反復という内側の実行ループを高速化するほど、人間の仕事は「何を価値ある仕事とみなすか」「どの証拠で安全と判断するか」「本番結果に誰が責任を持つか」という外側のループへ移ると論じています。Xでも「inner loop is capability, outer loop is engineering」という整理が、実務者向けの生産AIエージェント論として共有されています。
- なぜ面白いか:
  - 技術: ループを単なる自動リトライではなく、独立検証・証跡・承認境界を持つ制御系として設計する実務指針になっています。
  - 人文: これは認識論的には「モデルの自信」ではなく「提示された証拠」を知識の根拠にする態度であり、技術的権限と道徳的責任を分離しないための哲学です。人間は作業者から、判断・説明・責任の制度を保つ実践知の担い手へ移ります。

### 2. 日本語記事「ループエンジニアリングをどう始めるか」— 実践知としての反復設計

- 出典: Web記事（`<btn open />`） / Xでの日本語圏共有
- 日付: 2026-07-21
- リンク: https://www.btnopen.com/posts/lean-loop-engineering
- 要約: ボードゲーム用コンパニオンアプリ開発を例に、スマートフォンからタスクを投げ、AIエージェントに要件を磨かせ、PRをレビューしてマージするという実運用のループが説明されています。記事は、ループエンジニアリングを「コードを書く」行為の自動化ではなく、人間の仕事を設計・評価・判断へ寄せる変化として位置づけています。
- なぜ面白いか:
  - 技術: 実際のPR、レビュー、マージ、要件改善をつなぐ開発サイクルとして、ループがCI/CDやGCのような自動化史の延長に置かれています。
  - 人文: ここで重要なのは「職人技の消滅」ではなく、身体化された勘を評価軸・制約・レビュー手順へ外在化することです。暗黙知をループに埋め込む営みは、アリストテレス的な実践知（phronesis）をソフトウェア・プロセスへ翻訳する試みに見えます。

### 3. LEAF（Plan → Act → Observe → Verify → Remember → Repeat）— ループを知識生成装置として見る枠組み

- 出典: X投稿 / フレームワーク共有
- 日付: 2026-07-19〜2026-07-22頃（X検索結果上の直近議論）
- リンク: https://x.com/ajay4ai/status/2078214122295222495
- 要約: LEAFは、Plan、Act、Observe、Verify、Remember、Repeatを一連のループとして整理し、AIエージェントを長いプロンプトではなく、目標・ツール実行・独立検証・外部記憶・監査を備えたソフトウェアシステムとして扱う枠組みです。Plan-Execute-Verify、Maker-Checker、Test-Driven Loop、Compiler-Driven Loopなどの実装パターンも同じ思想に接続されます。
- なぜ面白いか:
  - 技術: 成功条件、検証器、状態ファイル、停止条件、監査ログをループの部品として明示するため、長時間エージェント運用の失敗モードを扱いやすくなります。
  - 人文: 認識論的には、これは「生成された文」を知識とみなすのでなく、観察・検証・記憶を経て正当化されたものだけを次の行為の前提にする方法です。サイバネティクスの sensor → controller → actuator → feedback を、AI開発の実践に再移植している点が面白いです。

### 4. Claude Code「ループ」分類の日本語圏での再解釈 — 自律性を段階として設計する

- 出典: X投稿 / 公式ドキュメント参照
- 日付: 2026-07-09以降に継続的に共有（公式ドキュメントは現行ページ）
- リンク: https://docs.anthropic.com/en/docs/claude-code/common-workflows
- 要約: X上では、Claude Codeのループ実践が turn-based、goal-based、time-based、proactive という4類型で整理され、特に日本語圏では「AIに丸投げするのではなく、何を任せるか、いつ止めるか、どう検証するか、コストをどう抑えるか」の設計論として受け取られています。関連投稿では、Lighthouseスコア、CI成功、最大試行回数など、検証可能な停止条件が強調されています。
- なぜ面白いか:
  - 技術: ループの開始条件と終了条件を人間・目標・時間・イベントで分けることで、エージェント自律性を一気に上げず段階的に導入できます。
  - 人文: これは自由意志や自律性を「完全な独立」としてではなく、環境・制約・監督・目的の配置から生まれる度合いとして考える立場に近いです。実務的には、任せることは手放すことではなく、関係の設計を精密にすることだと示しています。

### 5. 複雑適応系・第三次サイバネティクスとしてのAIループ — 観察者とシステムの境界が溶ける

- 出典: X投稿 / 哲学・システム論的議論
- 日付: 2026-07-21〜2026-07-22頃
- リンク: https://x.com/PraCha98/status/2079481857121669526 / https://x.com/Mehrland/status/2079696706254307665
- 要約: 一部の議論では、AIエージェントは行為、フィードバック、状態更新、次の行為を通じて振る舞いが変化する複雑適応系として捉えられています。さらに第三次サイバネティクスの言葉では、人間がAIを観察するだけでも、AIが人間を代替するだけでもなく、人間とAIが相互観察・相互修正する一つのフィードバックループになるという見方が提示されています。
- なぜ面白いか:
  - 技術: マルチエージェントや長期運用の設計では、単体モデル性能よりも、状態・制約・観測・フィードバックの安定性がシステム品質を左右します。
  - 人文: これはウィーナー、アシュビー、フォン・フェルスター以来のサイバネティクスを、エージェント時代の共同作業へ読み替える動きです。人間は外部の観察者であり続けられず、AIが出す候補・失敗・証拠によって自分の判断基準も更新されるため、ループは技術装置であると同時に自己形成の場になります。
