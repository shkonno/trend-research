## 今日の一言
AI/LLM/Agent時代のDDDは「古い設計手法」ではなく、曖昧な業務世界をAIに渡せる形へ翻訳するための共通言語として再評価されています。

## トップ5

### 1. AI時代にDDDを“上流の生存スキル”として読み直すスレッド
- 出典: X投稿（Aaron Stannard / @Aaronontheweb）
- 日付: 2026-07-22
- リンク: https://x.com/Aaronontheweb/status/2079779798344192086
- 要約: Aaron Stannard氏は、AIが実装を高速化しても、複雑な業務文脈を集め、正確なドメインモデルへ変換し続ける人間の能力は代替されにくいと論じています。DDDの考え方を、LLMに渡す「定義可能な構成要素とフロー」を作る技術として読み替えている点が高シグナルです。
- なぜ面白いか:
  - 技術: 境界づけられたコンテキスト、ユビキタス言語、型で不正状態を表現不能にする発想が、LLMの入力コンテキストと生成結果の検証可能性を高める設計原則として再接続されています。
  - 人文: ここでのDDDは単なる設計パターンではなく、業務の現実を誰がどう名づけるのかという組織文化の問題です。AI時代のエンジニア像が「コードを書く人」から「現実をモデル化し、共同体の言葉を整える人」へ移ることを示しています。

### 2. Martin Fowler経由で広がる「LLMにはDSLと強い抽象が必要」という議論
- 出典: X投稿（Martin Fowler / @martinfowler）
- 日付: 2026-07-14
- リンク: https://x.com/martinfowler/status/2077023024155422927
- 要約: Martin Fowler氏が、Unmesh Joshi氏の記事を紹介し、LLMに高速だが誤ったコードを書かせないためにはDSLや強い抽象、明確な境界が必要だという議論が広がっています。DDDのユビキタス言語やドメイン固有語彙が、LLM時代の「生成を縛るレール」として見直されています。
- なぜ面白いか:
  - 技術: DSLはプロンプトよりも強い制約を持つインターフェースになり、LLMの自由生成をドメイン語彙・型・ワークフローへ閉じ込めるハーネスとして機能します。
  - 人文: DSL化とは、組織が何を重要な概念として認めるかを明文化する行為でもあります。言葉の整備が権限、責任、仕事の境界を再編するため、DDDは文化設計の方法論としても読めます。

### 3. Thoughtworksの「AI agent skillsはAnti-Corruption Layerではない」論点
- 出典: X投稿（Thoughtworks）
- 日付: 2026-07-21
- リンク: https://x.com/thoughtworks/status/2079474689861112120
- 要約: Thoughtworksは、AIエージェントの“skills”を既存システムとの反腐敗層そのものと見なすのは危険で、DDDに基づくエンタープライズAIアーキテクチャが必要だと紹介しています。エージェントを便利な接着剤として扱うほど、上流システムとの結合や意味の漏れが問題化する、という実務的な警告です。
- なぜ面白いか:
  - 技術: Anti-Corruption Layerをエージェントのツール呼び出しに矮小化せず、ドメインモデル変換・契約・境界の保護として設計する必要性を示しています。
  - 人文: AIエージェント導入は技術的自動化であると同時に、部門間の責任境界を揺らす組織変化です。どの言葉をそのまま通し、どの言葉を翻訳するかが、組織の摩擦を減らすか増やすかを決めます。

### 4. 日本語圏での「AIとドメイン駆動開発は相性がよい／ユビキタス言語が重要」投稿群
- 出典: X投稿（日本語圏の開発者投稿）
- 日付: 2026-07-15〜2026-07-22
- リンク: https://x.com/ababupdownba/status/2079908402818617677 / https://x.com/polinomii/status/2077329910667047268
- 要約: 日本語圏では、AIにUIや実装を生成させると奇妙な命名や抽象がコードベースへ入り込みやすいため、早い段階でユビキタス言語を調整・固定する必要がある、という実感ベースの投稿が見られました。AIとドメイン駆動開発の相性を肯定しつつ、現場語彙の作成が重要だという方向に収束しています。
- なぜ面白いか:
  - 技術: 命名・モデル・画面文言をLLM任せにせず、ドメイン語彙をプロンプト、設計ドキュメント、コード規約へ反映することで、生成コードの一貫性を保ちやすくなります。
  - 人文: ユビキタス言語は、単に英語風のクラス名を整えることではなく、現場の人が自分たちの仕事をどう語るかを尊重する実践です。日本語の業務語彙をAIにどう渡すかは、ローカルな組織文化をソフトウェアに残す問題でもあります。

### 5. 「LLMコミュニティがDDDを再発見する」ミームと、実装高速化後の設計回帰
- 出典: X投稿（ParanoidChip、liquidated0x、Aaron Stannard氏のフォローアップなど）
- 日付: 2026-07-22
- リンク: https://x.com/ParanoidChip/status/2079790555400417781 / https://x.com/liquidated0x/status/2079809033804906597 / https://x.com/Aaronontheweb/status/2079924442126291423
- 要約: 「LLM界隈はDDDを再発見するのでは」「DDDは timeless、AIは実装を速くするだけ」といった反応が出ており、vibe codingから構造化された設計・仕様・境界づけへ関心が戻っています。Aaron氏のフォローアップでは、ユビキタス言語、垂直スライス、イベント駆動、TDD/BDDがLLMにとっての推論しやすい足場として整理されています。
- なぜ面白いか:
  - 技術: 実装速度が上がるほど、要求・境界・イベント・状態遷移の曖昧さがボトルネックになり、DDDやイベントストーミング的な分解がAI開発ライフサイクルの前提になります。
  - 人文: “再発見”という語は、技術コミュニティが新しい道具のたびに古い知恵を忘れ、別名で取り戻す歴史を示しています。DDDの復権は、効率化の先に残る人間的な仕事が「意味の交渉」であることを思い出させます。
