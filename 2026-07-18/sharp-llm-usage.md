## トップ5

### 1. SearchOS-V1: 失敗記憶と証拠グラフで検索エージェントのループを止める
- 出典: arXiv
- 日付: 2026-07-16
- リンク: http://arxiv.org/abs/2607.15257v1
- 要約: Web検索型エージェントが長い履歴の中で進捗を見失い、同じ失敗検索を繰り返す問題に対して、Frontier Task、Evidence Graph、Coverage Map、Failure Memoryとして進行状態を明示化する研究。情報収集を「会話」ではなく、引用付きのリレーショナルな表埋め問題として扱う点が実践的です。
- なぜ面白いか:
  - 技術: プロンプト履歴に暗黙の状態を押し込まず、探索範囲・証拠・未完了・失敗を構造化メモリとして持たせる設計は、調査エージェントの品質保証にそのまま応用できます。
  - 人文: 調査とは「答えを出す」だけでなく、どこを探し、何が見つからず、なぜ次へ進むのかを説明する営みです。SearchOS的な設計は、AIに探偵役を任せるときの説明責任を人間が読み取れる形に戻します。

### 2. Bad Codex bug: フルアクセスのコーディングエージェントはファイル削除まで失敗する
- 出典: Simon WillisonによるThibault Sottiaux発言の紹介
- 日付: 2026-07-16
- リンク: https://simonwillison.net/2026/Jul/16/bad-codex-bug/
- 要約: GPT-5.6/Codexが、フルアクセスかつサンドボックスや自動レビューなしの条件で、意図せずファイル削除に至る事例を紹介。原因として、古い「適用対象パス」が誤って再利用され、パッチが別ファイルへ当たるような失敗モードが示されています。
- なぜ面白いか:
  - 技術: コーディングLLMの実運用では、モデル性能よりも権限境界、サンドボックス、自動レビュー、パッチ適用前の差分検証が最後の安全装置になります。
  - 人文: 「賢い助手」に全権を渡すと、失敗は創造的ミスではなく組織的事故になります。人間が責任を負えるワークフローにするには、信頼ではなく可逆性と監査可能性を設計する必要があります。

### 3. Libretto PR Agents: Playwrightの失敗を、その場でAIがPR化して直す
- 出典: Libretto / Hacker News Show HN
- 日付: 2026-07-16
- リンク: https://libretto.sh/debug-agents
- 要約: 既存のPlaywrightスクリプトに`debugFailure()`を足すだけで、失敗時にライブページを調査し、修正案をGitHub PRとして開く仕組み。実行時の意思決定をLLMに丸投げせず、通常の自動化は決定的スクリプトで動かし、壊れたときだけエージェントが修理する分離が鋭いです。
- なぜ面白いか:
  - 技術: ブラウザ自動化の本番フローをLLMで置換するのではなく、失敗パスにだけLLM診断とPR生成を差し込むため、コスト・速度・レビュー可能性のバランスが良いです。
  - 人文: 自動化の現場で本当に苦しいのは「毎日動くこと」より「壊れた理由を追うこと」です。Librettoは、人間の保守疲れを減らしつつ、最終判断をPRレビューという社会的儀式に残しています。

### 4. Kote: AIチャットとGit履歴を“後から使える開発文脈”として保存する
- 出典: Kote GitHub / Hacker News Show HN
- 日付: 2026-07-12
- リンク: https://github.com/pedroaugusto04/Kote
- 要約: Antigravity、Codex、Claude Code、OpenCodeなどのAIチャット、Git push、開発メモを自動取得し、検索可能な知識として再利用する開発者向けメモリ層。LLMとの会話で解いたデバッグ、設計判断、暗黙知を、数週間後のPRや調査で再利用する発想です。
- なぜ面白いか:
  - 技術: コンテキストエンジニアリングを「毎回プロンプトに貼る作業」から、会話・コミット・ノートを横断する検索可能な外部記憶へ移しています。
  - 人文: ソフトウェア開発の知識は、コードそのものより「なぜそうしたか」に宿ります。Koteのような道具は、AI時代に散逸しやすい判断の物語を、チームが再び読み返せる記録に変えます。

### 5. Better Models: Worse Tools — 新しいClaudeほどツール呼び出しが壊れる事例
- 出典: Armin Ronacherブログ
- 日付: 2026-07-04
- リンク: https://lucumr.pocoo.org/2026/7/4/better-models-worse-tools/
- 要約: 新しいClaudeモデルで、特定のエージェント履歴を経たあとにツール呼び出しJSONへ余計な内容が混入し、編集系ツールが失敗する回帰を報告。単発プロンプトでは再現せず、ファイル読解・診断・複数行編集のような実運用の長い文脈で発生する点が重要です。
- なぜ面白いか:
  - 技術: LLM評価は単発ベンチでは足りず、長いエージェント履歴、複数ツール、編集差分、再試行を含む“作業軌跡”で検証する必要があります。
  - 人文: 道具の進歩は直線的ではなく、ある能力が伸びると別の信頼性が落ちることがあります。人間は「最新版だから安心」という物語から離れ、道具ごとの癖を観察する職人性を取り戻す必要があります。
