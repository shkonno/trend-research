### 1. 仕様書は最新に保てない。では何が仕様の正か
- タイトル: 仕様書は最新に保てない。では何が仕様の正か
- 出典: Qiita記事
- 日付: 2026-07-14
- リンク: https://qiita.com/ikeyansaza/items/d96867ba05e337f2e3ac
- 要約: AI駆動開発では散文仕様がすぐ古びるため、CIで同期が強制されるテストを現行仕様の正とし、ADRとユビキタス言語だけを長寿命の手書き文書として残す、という整理。ユビキタス言語を「チーム内の認識合わせ」から「コンテキストを共有しない実装者、すなわちAI/エージェントへの入力」へ格上げしている点が重要です。
- なぜ面白いか:
  - 技術: テスト、ADR、ユビキタス言語をそれぞれ異なる更新頻度と検証可能性で分離し、LLMに渡すコンテキストの腐敗を抑える設計論になっています。
  - 人文: DDDの言語観が、人間同士の共同理解から、人間と非人間エージェントの間で共有される「作業の公共語」へ広がっています。ドキュメントを信じる/疑うという組織文化の問題を、CIという制度設計に接続しているのも興味深いです。

### 2. Archally Blueprint Schema
- タイトル: Archally Blueprint Schema
- 出典: GitHubリポジトリ
- 日付: 2026-07-17更新
- リンク: https://github.com/Archally/blueprint-schema
- 要約: ドメイン設計、意思決定記録、ビジネスルール、ガバナンス、アーキテクチャを単一のYAMLスキーマにまとめ、OpenAPI、AsyncAPI、BPMN、UML、PRD、イベントストーミングボード、MCP経由のAIエージェント接地に使うというプロジェクト。READMEでは、設計知識がWiki、スライド、ホワイトボード写真、Confluence、部族的記憶に分散する問題を「fragmented truth」と表現しています。
- なぜ面白いか:
  - 技術: DDD/イベントストーミングの成果物を、生成AIが参照できる機械可読なシステム地図へ変換する試みです。
  - 人文: 「アーキテクチャ知識は誰の記憶に宿るのか」という組織人類学的な問題に踏み込んでいます。暗黙知を奪うのではなく、地図として共有することで、新参者やAIエージェントも同じ地形を歩けるようにする発想が面白いです。

### 3. AI時代のシステム崩壊に備えよ：仕様・ドメイン・イベントの「3つの駆動」で構築する究極のアーキテクチャ
- タイトル: AI時代のシステム崩壊に備えよ：仕様・ドメイン・イベントの「3つの駆動」で構築する究極のアーキテクチャ
- 出典: Qiita記事
- 日付: 2026-07-12
- リンク: https://qiita.com/satouo/items/1e39fd9e24e2961c2710
- 要約: AIによる高速コーディングが、エラー隠蔽、コピペ、技術的負債、安定性低下を招く危機に対し、ドメイン駆動、イベント駆動、仕様駆動を組み合わせる提案。DDDで業務境界を切り、EDAで障害伝播を抑え、SPEC.md/AGENTS.mdと複数エージェント監査で幻覚やズルを検出する構図です。
- なぜ面白いか:
  - 技術: DDDをLLMのコンテキスト制限と障害局所化のための境界設計として使い直し、イベント駆動やマルチエージェント監査と接続しています。
  - 人文: 「AIは速いが、責任を持たない短期契約者のように負債も作る」という比喩が、組織文化の緊張をよく表しています。速度崇拝から、監査・境界・合意を重んじる文化への揺り戻しとして読めます。

### 4. ai-refinement-method
- タイトル: ai-refinement-method
- 出典: GitHubリポジトリ / プロジェクトサイト
- 日付: 2026-06-24更新（14日超過だが関連性が高いため採用）
- リンク: https://github.com/nlawstudio/ai-refinement-method
- 要約: 「vibe codingではなくvibe spec'ing」を掲げ、自然言語の意図をイベントストーミング、DDD、リファインメント、脅威モデリング、受け入れ条件、失敗するテストへ変換してから、Claude Code/Cursor/Codexなどの実装ツールへ渡す方法論。コードが安くなった時代には、仕様品質こそがボトルネックになるという主張が中心です。
- なぜ面白いか:
  - 技術: LLMを実装者ではなく上流設計のファシリテータとして使い、イベントストーミングとDDDを「ビルド可能な仕様」に落とすパイプラインを提示しています。
  - 人文: 設計会議でしか生まれなかった対話・保留・合意形成を、エージェント時代の反復可能な儀式へ変換しようとしています。人間の曖昧な意図を急いでコード化せず、まず共同で名前を付ける態度がDDDらしいです。

### 5. Automating Domain-Driven Design: Experience with a Prompting Framework
- タイトル: Automating Domain-Driven Design: Experience with a Prompting Framework
- 出典: arXiv論文
- 日付: 2026-03-27（古いが、DDD×LLMの直接的な研究として採用）
- リンク: http://arxiv.org/abs/2603.26244v1
- 要約: DDDを、ユビキタス言語の確立、イベントストーミングのシミュレーション、境界づけられたコンテキストの特定、集約設計、技術アーキテクチャへのマッピングという5段階に分解し、LLMプロンプティングで支援する研究。ケーススタディでは前半の成果物は有用だった一方、後半では小さな誤りが累積し、完全自動化には向かないと報告しています。
- なぜ面白いか:
  - 技術: LLMはDDDの初期探索や会話のたたき台には強いが、集約や技術アーキテクチャの精密化では専門家レビューが不可欠だと実証的に線引きしています。
  - 人文: 「自動化」ではなく「協働するスパーリングパートナー」という結論が、DDDの本質である対話性を守っています。AIが設計者を置き換えるのではなく、議論すべきトレードオフを浮かび上がらせる道具になる、という慎重な文化観が示されています。
