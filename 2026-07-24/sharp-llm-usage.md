### 1. SANS「LLM Usage Cheat Sheet」：実務LLM利用をセキュリティ視点で整理
- 出典: SANS公式ページ / X投稿
- 日付: 2026-07-21（X投稿日。公式ページの日付はページ上で明示確認できず）
- リンク: https://www.sans.org/posters/llm-usage-cheat-sheet
- 要約: SANSのチートシートは、プロンプト、AI支援ワークフロー評価、MCP、コンテキストエンジニアリング、エージェントワークフロー、AIセキュリティリスクを横断して扱う実務リファレンス。X上でも、単なる「便利プロンプト集」ではなく、過剰な自律性、プロンプトインジェクション、評価、運用設計まで一枚で俯瞰できる資料として共有されていた。
- なぜ面白いか:
  - 技術: LLM活用を「入力文の工夫」ではなく、ツール連携、検証、権限、攻撃面、評価を含むシステム設計問題として扱っている。
  - 人文: LLMを仕事に入れることは、単に個人の生産性を上げる話ではなく、組織の責任境界を作り直す行為でもある。セキュリティ文脈のチートシートは、人間がAIにどこまで任せるかという統治の問いを実務の言葉に翻訳している。

### 2. Prompt / Context / Loop / Harnessの4層スタック：失敗原因を層で切り分ける
- 出典: explainx.aiブログ / X上の関連議論
- 日付: 2026-06-29公開、2026-07-13更新（重要だがやや古いものとして採用）
- リンク: https://explainx.ai/blog/context-prompt-loop-harness-engineering-stack-2026
- 要約: 記事は、プロンプト、コンテキスト、ループ、ハーネスを同義語として混ぜず、別々の設計層として整理する。たとえば「出力が悪い」時にプロンプトを書き換えるのか、RAGや履歴剪定を直すのか、停止条件を変えるのか、リトライや検証コードを直すのかを切り分ける診断フレームとして使える。
- なぜ面白いか:
  - 技術: 障害対応の単位を、単発メッセージ、モデルに見せる情報、反復ワークフロー、実行基盤に分解できるため、LLM活用のデバッグ精度が上がる。
  - 人文: これは「AIにどう頼むか」から「人間と機械の作業制度をどう設計するか」への転換である。チーム内でこの層を共有すると、魔法のプロンプト探しではなく、共同作業の責任分担としてAI利用を語れる。

### 3. Cloudflare Workers AIのPrompt Caching：モデル活用にもキャッシュ設計が必要
- 出典: Cloudflare公式ドキュメント / X上のモデルルーティング議論
- 日付: 2026-04-21更新（古いが、2026-07-11のX投稿でモデル親和性・キャッシュ文脈として再注目）
- リンク: https://developers.cloudflare.com/workers-ai/features/prompt-caching/
- 要約: Cloudflareは、同じ接頭辞を持つプロンプトのprefillを再利用するprefix cachingを説明し、`x-session-affinity`で同一セッションを同じモデルインスタンスに寄せる方法を示している。静的なシステムプロンプトやツール定義を前に置き、時刻やユーザー固有情報のような変動要素を後ろに置く、という具体的なプロンプト構造の作法が重要になる。
- なぜ面白いか:
  - 技術: LLMワークフローのコストと速度は、モデル選択だけでなく、コンテキストの順序、セッション固定、ルーティング方針で大きく変わる。
  - 人文: 「賢く聞く」だけではなく「何を変えずに保つか」を設計する発想が必要になっている。これは、人間の仕事でも反復可能な型や文脈を整えるほど創造的な余白が増える、という古典的な知恵に近い。

### 4. arXiv「AI Agents Do Not Fail Alone: The Context Fails First」：失敗はエージェントだけでなく文脈から生まれる
- 出典: arXiv
- 日付: 2026-07-15
- リンク: https://arxiv.org/abs/2607.14275
- 要約: この論文は、エージェントの失敗をモデル単体の問題ではなく、命令、ツール、メモリ、取得知識、ガードレール、信頼できない入力を含む「コンテキスト品質」の問題として測定する。ProofAgent-Harnessで、役割明確性、ガードレール、命令整合性、ツールスキーマ品質、根拠十分性、インジェクション耐性、トークン効率の7基準を評価する。
- なぜ面白いか:
  - 技術: 行動結果を見る前にコンテキスト品質を独立指標として測るため、幻覚、ツール誤用、制約無視、トークン浪費の予兆検知に使える。
  - 人文: 「AIが失敗した」と言うだけでは、環境を作った人間側の責任が見えなくなる。この論文は、失敗を個体の能力不足ではなく、制度・文脈・道具立ての産物として読む視点を与える。

### 5. 日本語Xでのn8n中心のAIワークフロー実践：LLMを業務の細い管に通す
- 出典: X投稿（日本語）
- 日付: 2026-07-23
- リンク: https://x.com/yasukisabu/status/2080133388468736472
- 要約: 日本語圏では、n8nを使って社内SlackボットのRAG化、問い合わせ分類、CRM格納、Web収集から競合分析レポート生成までを組む実践が目立った。MakeやZapier的な単純自動化から、コードノードでJSON整形、エラーフォールバック、RAGを組み込む高頻度・複雑なLLMワークフローへ関心が移っている。
- なぜ面白いか:
  - 技術: LLM出力をそのまま使うのではなく、ローコード基盤、コードノード、RAG、フォールバック、分類器をつないで「壊れても戻れる」業務パイプラインにしている。
  - 人文: 日本語の現場事例は、AI導入が大企業の研究開発だけでなく、日々の問い合わせ、整理、報告という地味な労働に浸透していることを示す。LLM活用の鋭さは派手なデモより、失敗しやすい実務の継ぎ目をどれだけ丁寧に縫えるかに現れる。
