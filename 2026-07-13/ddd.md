## 今日の一言
AI/LLM時代のDDDは、単なる実装パターンではなく、エージェントに業務境界・言葉・責任を渡すための「社会的インターフェース」として再注目されています。

## トップ5

### 1. Domain-Driven Design in Practice: A Large-Scale Empirical Characterisation of the Open-Source Ecosystem
- 出典: arXiv
- 日付: 2026-07-07
- リンク: https://arxiv.org/abs/2607.06471v1
- 要約: GitHub上のDDD関連リポジトリ11,742件候補から、GPT-4oの多数決検証を含むセマンティック検証で2,502件を抽出し、実践状況を大規模に特徴づけた研究。DDD採用は2017年以降に急伸し、C#とTypeScriptが実践面で目立つ一方、25.3%のプロジェクトで明示的なビジネス文脈が記録されていない点を指摘しています。
- なぜ面白いか:
  - 技術: DDDを「語られる設計論」から、リポジトリ規模・言語・アーキテクチャ・持続性で測れる実証対象へ移した点が重要です。
  - 人文: DDDの核心であるドメイン意図がコード履歴に残らない問題は、組織の記憶喪失そのものです。AIエージェントがコードを増やすほど、なぜその境界と言葉が選ばれたかを保存する文化が問われます。

### 2. AI時代のシステム崩壊に備えよ：仕様・ドメイン・イベントの「3つの駆動」で構築する究極のアーキテクチャ
- 出典: Qiita
- 日付: 2026-07-12
- リンク: https://qiita.com/satouo/items/1e39fd9e24e2961c2710
- 要約: AIコーディングの加速が、エラー隠蔽・コピペ・短期コードチャーンを増やし得るという問題意識から、DDD・イベント駆動・仕様駆動を組み合わせる設計論を提示しています。DDDを、LLMに巨大な全体仕様を渡さず、境界づけられたコンテキスト単位で制御するための実践として位置づけています。
- なぜ面白いか:
  - 技術: bounded contextをLLMのコンテキスト長・Lost in the Middle・越権実装への対策として再解釈している点が実務的です。
  - 人文: 「AIに現実世界の境界線を教える」という表現が象徴的で、DDDが人間同士の合意形成だけでなく、人間と機械の協働規範になりつつあります。速度礼賛へのブレーキとして、組織文化の成熟も求めています。

### 3. Tandem: A shared seat inside the engineer's coding-agent session
- 出典: GitHubリポジトリ
- 日付: 2026-07-13更新
- リンク: https://github.com/mherzog4/tandem
- 要約: Claude Code、Codex、Gemini、Aiderなどの coding agent セッションを、非技術ステークホルダーと共有するためのCLIツール。READMEでは、ステークホルダーが自分の言葉でプロンプトに意図を書き、エンジニアが実行承認と技術的枠組みを担い、EventStorming風のDomain Boardが共有語彙を `DOMAIN.md` に記録すると説明されています。
- なぜ面白いか:
  - 技術: EventStormingとユビキタス言語を、AIエージェントに読ませる永続コンテキストへ接続している点が新しいです。
  - 人文: DDDの「会話からモデルを作る」実践が、プロンプトを書く場そのものへ移動しています。非技術者の言葉を翻訳しすぎずに残す設計は、AI時代の権限分配と参加型設計の問題にもつながります。

### 4. golden pathテンプレートに「受注-在庫」機能をAIエージェントで実装させ、実AWSで検証してみた
- 出典: Qiita
- 日付: 2026-07-13
- リンク: https://qiita.com/itouhi/items/98a18cf43e280308c965
- 要約: FastAPI、Vue 3、Terraform、3層品質ゲート、SDD運用を含む golden path テンプレートで、Claude Codeに受注・明細・在庫・在庫引当の業務機能を実装させ、実AWS環境で検証した記録です。単一CRUDでは表現できない複数モデル横断トランザクションを扱い、要件定義、基本設計、タスク分解、PR、E2E確認までを追っています。
- なぜ面白いか:
  - 技術: DDDそのものを前面に出す記事ではありませんが、受注・在庫・在庫引当という典型的なドメイン不変条件をAIエージェントに実装させる検証として価値があります。
  - 人文: AIに「丸投げ」せず、破壊的操作前の確認や要件承認を挟む運用は、責任を人間側に残す設計です。業務知識を手順化するだけでなく、誰が判断し誰が承認するかという組織儀礼も見えます。

### 5. コード生成ルールとしての憲法と哲学
- 出典: Qiita
- 日付: 2026-07-08
- リンク: https://qiita.com/atatago/items/4685d2f224d7a3f23272
- 要約: Claude CodeなどのLLMコード生成に対し、場当たり的なガードレールではなく、システム憲法、プログラミング哲学、法律、細則という文書階層を設けて、自律的で一貫した生成を目指す提案です。哲学の項目にDomain-Driven Design、オブジェクト指向、開閉原則、簡潔性、テスト容易性などを置いています。
- なぜ面白いか:
  - 技術: DDDを個別パターンではなく、AIが設計判断を行う際の上位原則・制約体系に組み込む発想が参考になります。
  - 人文: 「憲法」「法律」「細則」という比喩は、コードベースを小さな共同体として扱う見方です。AIエージェントに守らせる規範を作ることは、チームの価値観を明文化する政治的・文化的作業でもあります。
